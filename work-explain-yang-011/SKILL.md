# work-explain-yang-011 · 文档人话翻译

## 目标

将"公司文档""跨部门业务沟通"等专业、绕口、抽象的表达，转译为人人能懂的「人话」，同时保留所有关键语义单元，不遗漏、不虚构、不重复。

---

## 执行流程

### Step 1：文档拆分

读取文档后，按语义将文档拆分为小颗粒度的 **part**（part1, part2, ... partN）。

- 拆分依据：语义边界（主题切换、说话人切换、场景切换等），而非段落或页码。
- 每个 part 应包含一个独立、完整的语义块。
- 如果文档是表格（Excel）、PDF 扫描件、或 PPT，先提取可读文本再拆分。

### Step 2：板块校验

逐 part 检查是否存在：
- **遗漏**：原文中的信息在 part 中没有对应。
- **重复**：同一语义在多个 part 中出现。
- **无中生有**：part 中出现了原文不存在的断言、数据、或结论。

发现以上问题时，标注并修正，输出修正后的 part 列表。

### Step 3：逐个 part 要素分析

对每个 part，提取以下要素（有则填，无则跳过）：

#### 基本表达要素

| 要素 | 说明 |
|------|------|
| **谁** | 行为主体 / 责任方 / 角色 |
| **说了什么** | 口头或书面表达的内容 |
| **做了什么** | 已执行的动作 / 已发生的事件 |
| **需要什么** | 诉求 / 依赖 / 资源需求 |
| **有什么数据** | 数字 / 指标 / 统计 |
| **截止日期** | 时间节点 / deadline |

#### 基本表达关系

- **人与人的关系**：谁对谁汇报、谁与谁协作、谁批准谁
- **人与事的关系**：谁负责什么事、谁被什么事影响
- **人与物/工具的关系**：谁用什么系统、谁管理什么资源

### Step 4：人话输出

将分析结果转化为自然、通顺的中文人话输出。

**强制格式规则：**

- 每句话中，**必须**使用 `<>`、`()`、`「」`、`{}` 对重要语义单元进行框定强调。
- 使用规则（优先级从高到低）：
  - `<>` — 核心概念 / 抽象术语（如：`<无明>`、`<OKR>`、`<SLA>`）
  - `()` — 补充说明 / 条件限定（如：`(截止至 Q3 末)`）
  - `「」`— 引用原文 / 专有说法（如：`「对齐拉通」`、`「颗粒度」`）
  - `{}` — 数据 / 指标 / 数字（如：`{35%}`、`{200 万}`、`{2026-06-15}`）

- 禁止长难句。单句不超过 40 字。长信息用多句分述。
- 每个 part 的输出以 `## Part N` 开头，方便对照。
- 全文末尾附一个「核心摘要」（≤ 5 条要点）。

**输出结构模板：**

```
## 人话版：《文档标题》

### Part 1 — [一句话概括本 part]
[人话内容，要素与关系自然嵌入，关键语义用符号框定]

### Part 2 — [一句话概括本 part]
[人话内容...]

...

### 核心摘要
1. ...
2. ...
3. ...
```

---

## 注意事项

1. **禁止遗漏**：原文中每个有信息量的语义单元都必须出现在输出中。
2. **禁止虚构**：不要添加原文没有的信息或推断。
3. **保留歧义**：原文确实含糊之处，用 `(原文未明确)` 标注，不擅自补充。
4. **专业术语保留**：不替换专业术语本身，而是在其后用框定符号 + 人话解释。例如：`<SLA>(服务等级协议，即承诺的响应速度)`。
5. **Excel/表格处理**：先用 `markitdown` 或 `xlsx-tool read` 提取文本内容，再执行上述流程。
6. **PDF 处理**：先用 `markitdown` 或 Read 工具提取文本，再执行上述流程。
